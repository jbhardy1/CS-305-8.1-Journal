# CS-305 Portfolio: Software Security
### This repository contains the Artemis Financial Practices for Secure Software Report from Project Two. This artifact demonstrates my ability to implement secure communications through encryption technologies, generate SSL certificates, deploy cryptographic hash algorithms, and perform vulnerability assessments using industry standard tools.

**Reflection**

  Artemis Financial is a financial consulting company that develops individualized financial plans for its customers, covering savings, retirement, investments, and insurance. The company wanted to modernize its operations and add a file verification step to its web application to ensure secure communications. Specifically, they needed a data verification mechanism in the form of a checksum, along with secure HTTPS communication to protect sensitive client financial data transmitted through their public web interface.
  When I identified the software security vulnerabilities, I think the strongest part of my work was the methodical approach I took to addressing each layer of the application. I started with the architecture review, then moved through cryptography, client/server configuration, error handling, and static testing in a deliberate sequence rather than trying to fix everything at once. Coding securely is important because a single vulnerability in a financial application can expose sensitive client data, create legal liability, and damage the trust that a company like Artemis Financial depends on. Software security adds direct value to a company's well-being by protecting its reputation, reducing the risk of costly data breaches, and demonstrating to clients and regulators that the organization takes data protection seriously.
  The most challenging part of the vulnerability assessment was working with the OWASP dependency-check tool. The original version specified in the project (5.3.0) could no longer connect to the NVD data feeds because NIST retired those endpoints, so I had to research the issue, upgrade to a current version, and configure an NVD API key to get the scanner working. That experience was also one of the most helpful parts of the project because it reinforced how quickly security tooling and infrastructure can change, and why staying current with your tools matters just as much as staying current with your code.
  I increased layers of security in several ways. I generated a self-signed SSL certificate using the Java Keytool with RSA at 2048 bits and configured the Spring Boot application to use HTTPS on port 8443, encrypting all data in transit. I implemented a SHA-256 checksum endpoint using the Java MessageDigest class to provide data integrity verification. I also added proper error handling in the controller so that exceptions would return safe messages rather than exposing internal application details. In the future, I would continue using the OWASP dependency-check tool for static analysis of third-party libraries, and I would also incorporate OWASP ZAP or similar dynamic analysis tools for testing running applications. For deciding which mitigation techniques to use, I would follow the vulnerability assessment process flow diagram and prioritize based on the CVSS severity scores reported by the scanning tools.
  To make certain the code and application were functional and secure, I compiled and ran the refactored application to verify it started without errors and that the /hash endpoint returned the correct SHA-256 checksum output over HTTPS. After refactoring, I ran the OWASP dependency-check tool a second time to confirm that my changes did not introduce any new vulnerabilities. I also performed a manual code review looking for syntactical errors, logical flaws, and security issues such as exposed stack traces or hardcoded credentials.
  The resources and tools I used that will be helpful going forward include the Java Keytool for certificate generation, the Java MessageDigest class for cryptographic hashing, the OWASP dependency-check Maven plugin for static vulnerability scanning, and the Spring Boot SSL configuration properties for enabling HTTPS. I also gained practical experience reading and interpreting CVE reports and understanding how to suppress false positives in dependency scanning, which is something that comes up regularly in professional development environments.
  From this assignment, I would show future employers the Practices for Secure Software Report along with the refactored code. Together, these artifacts demonstrate that I can assess a client's security requirements, recommend and implement appropriate cryptographic solutions, configure secure communications, use industry standard vulnerability scanning tools, and document my process clearly. The work shows both technical ability and the kind of structured, security-conscious thinking that organizations need from developers working on applications that handle sensitive data.
